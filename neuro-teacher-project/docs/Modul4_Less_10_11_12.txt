Когда одного агента недостаточно ↘️ 
Мы много можем сделать при помощи одного агента — изучить скиллы, подключить разные 
коннекторы в виде MCP. Тем не менее несколько агентов нам нужны вовсе не для красоты, а 
чтобы разделить ответственность, когда у нас есть разные типы задач: например, поиск 
информации, написание, редактура и валидация финального результата. 
Важно не количество агентов, а управляемость результата и то, что мы получаем на выходе. 
Если в ваших задачах один агент справляется быстро, выполняет конкретную задачу и даёт 
понятный, измеримый результат — усложнять не стоит. Но если задача приобретает 
характер многофазности и многоступенчатости, тогда можно задуматься о субагентах. 
2. Три модели взаимодействия агентов 
Последовательная модель ↘️ 
     
Последовательная модель — когда у нас есть задача, состоящая из определённых этапов, и 
каждый последующий этап зависит от предыдущего. Это может работать медленнее, чем другие 
мультиагентные системы, но это абсолютно рабочий и понятный подход. 
Пример последовательной модели: 
• Запрос пользователя 
• Агент-исследователь собирает факты 
• Агент-редактор превращает факты в структурированный текст 
• Агент-проверяющий ищет ошибки 
• Финальный результат 
Запрос пользователя

→ Агент-исследователь собирает факты 
→ Агент-редактор превращает вывод в понятный ответ 
→ Агент-проверяющий ищет ошибки 
→ Финальный результат 
Параллельная модель ↘️ 
     
Параллельная модель — задача разделяется на составные части, которые можно выполнять 
одновременно. Несколько субагентов работают параллельно и возвращают результаты главному 
агенту. Это позволяет работать быстрее. 
     
Параллельная модель чуть сложнее: результаты могут быть несогласованными. Необходим чёткий 
контракт — формат, в котором агенты получают задачи и возвращают результаты. 
Пример параллельной модели: 
• Один запрос 
• Агент А ищет официальное определение, агент Б — практические примеры, агент В — 
риски и ограничения 
• Все результаты возвращаются главному агенту или агенту-агрегатору 
• Финальный результат 
Запрос пользователя 
├─ Агент A: ищет официальное определение 
├─ Агент B: ищет практические примеры 
└─ Агент C: ищет риски и ограничения 
        ↓ 
Агент-агрегатор объединяет результат 
        ↓ 
Финальный ответ 
Ролевая модель ↘️ 
     
Ролевая модель — у каждого агента есть чётко заданная роль: ресёрчер (исследователь), 
аналитик, райтер (автор), ревьюер (проверяющий). Роли прописываются заранее. 
       
Лишние роли усложняют процесс без пользы — не нагромождайте агентов без необходимости. 

===== PAGE 4 =====

Три модели не исключают друг друга — они вполне могут существовать вместе. Например, агенты 
с определёнными ролями могут работать параллельно, а потом последовательно. Это нормальная 
практика в реальных системах. 
Manager / Orchestrator 
├─ Researcher: ищет факты и источники 
├─ Analyst: делает выводы 
├─ Writer: пишет понятный текст 
└─ Reviewer: проверяет точность и полноту 
Что такое субагенты в Claude Code ↘️ 
     
Субагент — это всего лишь Markdown-файл с таким же YAML frontmatter, как и в скиллах, который 
лежит внутри настроек Claude Code. У него есть: 
• имя и описание 
• системная инструкция, которая подгружается при запуске 
• при необходимости — список доступных инструментов и скиллов 
Субагенты удобно использовать для повторяемых ролей: ревьюер, тестировщик, исследователь, 
документатор, отладчик. 
Главный агент никуда не исчезает. Если мы стартуем работу через него, он координирует 
субагентов: самостоятельно их запускает, выдаёт задачи, получает ответы и собирает финальный 
результат. 
Настройки субагентов бывают двух типов: 
• Проектный уровень: .claude/agents/название-агента.md внутри папки проекта 
• Глобальный (пользовательский) уровень: ~/.claude/agents/название-агента.md 
Инструменты для субагентов ↘️ 
Каждому субагенту можно выдать ровно те инструменты, которые нужны для его роли: 
• Read + Search — агенту-ревьюеру, который только читает файлы и выносит вердикт 
• Read + Search + Bash — если нужны терминальные команды (запросы к серверу, 
тестирование кода) 
• Read + Search + Bash + Edit — если агент ещё и исправляет файлы 
       
Если вы пока не уверены в выборе инструментов — выбирайте «все инструменты». Это снимает 
головную боль на старте, хотя в некоторых задачах чуть менее безопасно. 
     
Правило least privilege: если агент только проверяет результат, не даём ему право редактировать 
файлы. Если агенту не нужны команды терминала, не даём доступ к Bash. 

===== PAGE 5 =====

Примеры уровней доступа: 
Read, Grep, Glob — агент может провести ревью без правки файлов. 
Read, Grep, Glob, Bash — агент может провести ревью и выполнить проверки через терминал. 
Read, Grep, Glob, Edit, Bash — агент может проверять результат и исправлять файлы. 
3. Первый запуск субагентов 
Запуск субагентов без предварительной настройки ↘️ 
Начнём с простого примера. Открываем терминал, переходим в рабочую папку и запускаем Claude 
Code. 
Первая команда — /agents. Она показывает запущенные субагенты и библиотеку всех доступных 
агентов на проектном и пользовательском уровнях. 
Важный момент: субагентов можно использовать сразу, без предварительной настройки. 
Достаточно в запросе указать, что нужно запустить двух отдельных субагентов на разные темы — 
и Claude Code запустит их параллельно, потому что такова логика запроса. 
Вы можете без лишних заморочек сразу начать работать с субагентами — просто скажите 
главному агенту, что нужно создать одного агента для такой-то задачи и второго для другой. 
 
Напиши Claude Code следующий запрос: 
Как работают субагенты в Claude Code? Запусти отдельного субагента-исследователя на эту 
задачу и запусти второго субагента на задачу: как связаны субагенты и скиллы внутри Claude 
Code.

Мы наблюдаем, как главный агент запустил двух субагентов параллельно, каждый использует

Вызов субагента и экономия контекста ↘️ 
Субагента можно вызвать двумя способами: 
• Эксплицитно (явно, прямо, без намёков) — указать @имя-агента прямо в запросе 
@codebase-explainer (agent) для @finance-card 
• Неявно — попросить главного агента выполнить задачу, и он сам решит, когда вызвать 
нужного субагента 
Найди в моей директории проект «Файнанс карт» и сделай код-ревью для этого сайта. 
Используй субагента. 
Зачем вообще делегировать задачи субагентам? Потому что это экономит контекстное 
окно главного агента. 
Когда контекст заполняется, модель выполняет auto-compaction — сжимает всю предыдущую 
сессию до тезисов и «забывает» детали. Если мы делегируем тяжёлую работу (например, чтение 
большого количества файлов или ресёрч) субагенту, то расходуем его контекст, а главный агент 
получает лишь финальный вердикт и остаётся «чистым». 
       
Очень полезно делать все ресёрчи через субагентов. Если нужно найти документацию, собрать 
новости или прочитать блогпосты — запустите субагента для этой задачи. Главный агент (особенно

5. Демонстрация работы субагента 
Демонстрация: контент-завод на мультиагентной архитектуре ↘️ 
Разберём реальный пример — контент-завод, который берёт новость и превращает её в контент 
для соцсетей. Архитектура построена на скилле-оркестраторе и четырёх субагентах. 
Скилл news2social v2 задаёт главному агенту роль оркестратора: 
• Принять один источник (ссылку или текст) 
• Параллельно запустить трёх субагентов: Content Writer, Image Generator, HyperFrames 
Agent 
• Собрать результаты в JSON-файл 
• Передать JSON агенту-Publisher для публикации в соцсети 
Субагент Content Writer получает системные инструкции из предыдущей версии скилла — 
практически без изменений. Его задача: превратить источник в JSON со схемой постов 
для Telegram, Instagram, X и TikTok. Результат сохраняется в папку posts/. 
Субагент Image Generator использует MCP fal.ai: формирует image-промпт, отправляет его в GPT 
Image 2, ждёт результата и сохраняет картинку. Важно: в инструкциях прописаны конкретные 
инструменты MCP — submit_job и check_job_status, чтобы агент не зависал. 
Субагент HyperFrames генерирует видеоролик. В его frontmatter прописан ключ skills с названием 
скилла HyperFrames — так субагент получает всю информацию о том, как работать с этим 
инструментом. 
Субагент Publisher работает на скилле Postis. Ему выданы только инструменты bash и read — он не 
может изменять файлы, только читать и отправлять команды. 
 
Весь пайплайн в действии: 
1. Мы отправляем ссылку главному агенту. 
2. Он параллельно запускает Content Writer, Image Generator и HyperFrames Agent в фоновом 
режиме. 
3. Получает от них результаты и собирает JSON. 
4. Передаёт результат Publisherагенту, который публикует пост в X. 
Мы просто скинули ссылку — и получили текст, картинку и видео, опубликованные в соцсети. 
     
В процессе демонстрации возникла ошибка: Content Writer создавал текст длиннее 200 символов 
для X. Агент сообщал об успехе, хотя на самом деле получал ошибку API. Это важный урок: 
коммуникацию между агентами нужно прописывать детально, включая форматы ошибок и 
критерии валидации. Можно добавить отдельный этап проверки длины или настроить хук.

6. Когда мультиагентность нужна — и как начать 
Критерии выбора: один агент или система ↘️ 
Три главных признака того, что стоит задуматься о мультиагентной системе: 
• Объём задачи — задача большая и требует разных шагов 
• Повторяемость — процесс запускается регулярно 
• Типы работы — задача требует разных компетенций и инструментов 
Если задача похожа на один короткий запрос — берём одного агента. Если задача похожа 
на процесс с разными этапами и компетенциями — появляется смысл в команде субагентов. 
     
Риск ошибок в мультиагентных системах велик — это нормально. Агенты могут возвращать 
формально «успешный» ответ при реальной ошибке, неэффективно расходовать контекст или 
создавать лишнее. Задача агентного инженера — это увидеть и итерационно поправить. 
Вопросы для проектирования своей системы ↘️ 
Прежде чем строить мультиагентную систему, ответьте на эти вопросы: 
• Что мы хотим получить на выходе? 
• Какие этапы нужны, чтобы получить этот результат? 
• Какие навыки и инструменты нужны для каждого этапа? 
• Какие этапы можно отдать отдельным агентам, а какие — главному? 
• Как один агент передаёт результат следующему? 
• Как проверяется качество финального результата? 
       
Для передачи результатов между агентами лучше всего использовать JSON — это чёткий контракт, 
который все агенты понимают одинаково. 
Рекомендации для старта ↘️ 
Начинайте с дуэта: главный агент + один субагент с конкретной ролью и ограничениями. 
Полезная последовательность: 
1. Сначала реализуйте задачу вручную через одного агента — так вы поймёте весь процесс и 
все input/output 
2. Перенесите эту логику на взаимодействие главного агента и субагента 
3. Итерационно улучшайте: уточняйте роли, форматы передачи результатов, критерии 
качества 
4. Следите не только за финальным результатом, но и за логами взаимодействия агентов

Для каждой роли важно зафиксировать: цель, вход, выход и критерий качества. 
Мультиагентная система нужна не всегда. Её смысл — разделить разные типы работы между 
ролями и усилить автоматизацию. Для старта достаточно дуэта. Главный критерий: стал ли 
результат качественнее, понятнее и стабильнее.

Словарь занятия ↘️ 
• Мультиагентная система — архитектура, в которой несколько AI-агентов взаимодействуют 
для решения сложной задачи 
• Оркестрация — процесс управления работой нескольких субагентов главным агентом 
• Главный агент (оркестратор) — агент, который координирует субагентов, выдаёт им 
задачи и собирает результаты 
• Субагент — Markdown-файл с системной инструкцией и настройками, который запускается 
внутри текущей сессии Claude Code 
• YAML frontmatter — блок метаданных в начале Markdown-файла, содержащий имя, 
описание, модель и инструменты агента 
• Последовательная модель — агенты выполняют задачи цепочкой, каждый следующий 
зависит от результата предыдущего 
• Параллельная модель — несколько агентов работают одновременно над разными 
частями одной задачи 
• Ролевая модель — каждому агенту заранее назначена чёткая роль (ресёрчер, аналитик, 
райтер, ревьюер) 
• Контракт агента — чёткий формат (обычно JSON), в котором агент принимает данные и 
возвращает результат 
• Контекстное окно — ограниченный объём памяти модели в рамках одной сессии 
• Auto-compaction — автоматическое сжатие истории сессии, когда контекстное окно 
заполняется 
• Inherit from parent — настройка модели субагента, при которой он использует ту же 
модель, что и главный агент 
• Project scope — область памяти агента, ограниченная текущим проектом 
• Dry run — тестовый запуск без реального выполнения финального действия (например, 
публикации) 
• Pipeline — последовательность шагов обработки данных от входа до результата

# AI News to Social Version 2 
Ты — главный агент-оркестратор. Твоя задача: принять один источник про ИИ, собрать из него готовый пакет для соцсетей, получить изображение, получить ролик и передать результат в 
публикацию через отдельных subagents. 
 
Ты не пишешь контент сам, не генерируешь изображение сам и не публикуешь сам. Ты только управляет пайплайном, проверяешь артефакты между шагами и собираешь финальный 
результат. 
 
## Архитектура 
 
Используй subagents из `.claude/agents`: 
 
```text 
ai-news-content-writer-v2 
ai-news-image-generator-v2 
ai-news-hyperframes-agent-v2 
ai-news-postiz-publisher-v2 
``` 
 
Роли: 
 

===== PAGE 16 =====

```text 
ai-news-content-writer-v2 
  -> читает источник и возвращает social-content/v2 JSON 
 
ai-news-image-generator-v2 
  -> читает тот же источник, сам формирует image_prompt, генерирует изображение через GPT Image 2 и возвращает image_prompt + image_url + local_image_path 
 
ai-news-hyperframes-agent-v2 
  -> читает тот же источник, генерирует ролик через HyperFrames и возвращает video_prompt + local_video_path 
 
ai-news-postiz-publisher-v2 
  -> получает финальный JSON с media/image_url/video и делает dry-run или публикацию через skill `postiz` 
``` 
 
## Вход 
 
Один источник: 
- статья; 
- новость; 
- короткий пост; 
- фрагмент рассылки; 
- ссылка плюс цитата или краткое описание. 
 
Дополнительные параметры, если пользователь указал: 
- платформы для публикации: `telegram`, `instagram`, `x`, `tiktok`; 
- режим публикации: `dry-run`, `publish now`, `schedule`; 
- дата и время публикации; 
- требования к изображению. 
- требования к ролику, если пользователь их указал. 
 
Если режим не указан, используй `dry-run` и не отправляй публикации без явного подтверждения пользователя.

## Порядок работы 
 
### 1. Прими источник 
 
прими источник от пользователя и отправь его субагентам . 
 
В subagent `ai-news-content-writer-v2` передавай: 
- URL или текст; 
- заголовок; 
- дату публикации, если известна; 
- список платформ, если пользователь ограничил публикацию; 
- требование вернуть валидный JSON и путь к файлу. 
 
В subagent `ai-news-image-generator-v2` передавай: 
- URL или текст; 
- заголовок; 
- slug или тему для имени файла; 
- требования к изображению, если пользователь их указал; 
- желаемый формат: `png`; 
- размер по умолчанию: `landscape_4_3`; 
- качество по умолчанию: `high`. 
 
В subagent `ai-news-hyperframes-agent-v2` передавай: 
- URL или текст; 
- заголовок; 
- дату публикации, если известна; 
- slug или тему для имени файла; 
- требования к ролику, если пользователь их указал; 
- требование вернуть путь к готовому MP4 и краткий video prompt / scene plan. 
 
### 2. Запусти content, image и HyperFrames subagents параллельно

После подготовки источника запусти одновременно: 
 
```text

``` 
 
`ai-news-content-writer-v2` отвечает за тексты и базовый `social-content/v2` JSON. 
 
Ожидаемый результат: 
 
```json

  "status": "ok", 
  "json_path": "posts/YYYY-MM-DD-news-slug-social-content.json", 
  "content": { 
    "schema_version": "social-content/v2", 
    "source": {}, 
    "image_prompt": "...", 
    "platforms": {} 
  }, 
  "notes": [] 
} 
``` 
 
Проверь: 
- файл создан в `posts/`; 
- JSON валиден; 
- есть `schema_version: "social-content/v2"`; 
- есть `image_prompt`; 
- есть `platforms.telegram.content`, `platforms.instagram.content`, `platforms.x.content`, `platforms.tiktok.content`, если пользователь не сузил список платформ. 

===== PAGE 19 =====

 
Если есть ошибки, верни задачу content subagent на исправление. 
 
`ai-news-image-generator-v2` отвечает за визуальную интерпретацию источника, генерацию `image_prompt` и создание изображения.

```json 
  "notes": []
``` 

Проверь, что изображение создано или доступно по URL. 
 
`ai-news-hyperframes-agent-v2` отвечает за видео по источнику. 
 
Ожидаемый результат:

```json 
  "status": "ok", 
  "video_prompt": "HyperFrames prompt or scene plan",

} 
``` 

===== PAGE 20 =====

 
Проверь, что ролик создан и локальный MP4 доступен. 
 
Если один из трех параллельных subagents завершился с ошибкой, останови пайплайн, верни ошибочный шаг на исправление или сообщи пользователю, что именно не получилось. Не 
запускай publisher, пока все три результата не готовы. 
 
### 3. Собери финальный JSON 
 
Когда все три параллельных subagents вернули результат, собери их данные в финальный JSON. За основу возьми JSON от `ai-news-content-writer-v2`, затем добавь данные от `ai-news-
image-generator-v2` и `ai-news-hyperframes-agent-v2`. 
 
Обнови JSON-файл: 
 
```json 
{ 
  "image_prompt": "Prompt for GPT Image 2", 
  "image_url": "https://...", 
  "local_image_path": "posts/YYYY-MM-DD-news-slug-social-image.png", 
  "video_prompt": "HyperFrames prompt or scene plan", 
  "local_video_path": "posts/YYYY-MM-DD-news-slug-social-video.mp4", 
  "media": [ 
    { 
      "path": "posts/YYYY-MM-DD-news-slug-social-image.png", 
      "source_url": "https://..." 
    }, 
    { 
      "path": "posts/YYYY-MM-DD-news-slug-social-video.mp4", 
      "type": "video" 
    } 
  ] 
} 
``` 
 

===== PAGE 21 =====

Если publisher позже загрузит изображение через skill `postiz` и получит Postiz URL, он может заменить или дополнить `media` и `platforms.<name>.media`. 
 
### 4. Запусти publisher subagent 
 
Передай в `ai-news-postiz-publisher-v2`: 
- путь к финальному JSON; 
- список платформ; 
- режим: 
  - `dry-run`, если пользователь не просил публиковать сразу; 
  - `publish`, если пользователь явно попросил отправить публикации; 
  - `schedule`, если пользователь указал дату и время; 
- путь к локальному изображению; 
- публичный URL изображения, если есть. 
- путь к локальному ролику, если есть. 
 
Ожидаемый результат: 
 
```json 
{ 
  "status": "ok", 
  "mode": "dry-run", 
  "json_path": "posts/YYYY-MM-DD-news-slug-social-content.json", 
  "platforms": ["telegram", "instagram", "x", "tiktok"], 
  "postiz_result": {}, 
  "notes": [] 
} 
``` 
 
### 5. Финальный ответ пользователю 
 
Коротко сообщи: 
- где лежит JSON;

- где лежит изображение;

- что сделал publisher: dry-run, scheduled или published; 
- какие платформы обработаны; 
- какие ошибки или ограничения остались. 
 
## Ограничения 
- Не переписывай тексты после content subagent, кроме технической правки JSON-структуры. 
- Не выходи за пределы рабочей директории проекта. 
- Если один из subagents вернул ошибку, останови pipeline и верни понятный отчет.

1. Деплой проекта: сервер, SSH, SSL, базовая безопасность 
Зачем нужен деплой ↘️ 
     
Деплой — это момент, когда проект переносится из локальной среды разработки в интернет и 
становится доступен всем пользователям. 
Локальный запуск удобен для разработки: мы пишем код, проверяем изменения, исправляем 
ошибки, экспериментируем. Сайт работает по адресу localhost и виден только нам на нашем 
компьютере. 
     
Задеплоенный проект — это уже сервис, размещённый на удалённом сервере, к которому могут 
обращаться другие люди. 
Деплой — это запуск проекта в среде, где на него может посмотреть весь интернет. 
     

===== PAGE 39 =====

Сервер — это условно отдельный компьютер, который постоянно включён, подключён к интернету 
и готов обслуживать запросы пользователей: принимать их и отдавать ответы. У пользователя есть 
адрес, по которому он открывает сайт, API-сервис, бота, ИИ-продукт или что-то ещё. 
Три варианта платформ для размещения проекта ↘️ 
Рассмотрим три основных варианта того, где может жить проект. 
1. Managed-платформы: Vercel, Railway, Netlify и др. 
Это удобные сервисы, которые работают в связке с GitHub и позволяют быстро разместить код 
проекта без дополнительных настроек: без ручной установки SSL/TLS-сертификатов и часто без 
отдельной настройки домена. Загрузили — работает. Как правило, есть бесплатный тир. 
• Vercel — оптимален для фронтенда и проектов на фреймворке Next.js. Бэкенды тоже 
возможны, но менее удобны. 
• Railway — более универсальный вариант: базы данных, бэкенды, фронтенды. 
• Netlify — удобен для статических сайтов, лендингов и фронтенд-проектов. 
2. Обычный хостинг 
Удобен для лендингов, визиток, сайтов на Vanilla HTML/CSS/JavaScript, а также для сайтов 
на PHP и CMS: например, WordPress. Подходит для простых сайтов без сложной серверной логики. 
3. VPS (Virtual Private Server) 
Отдельный виртуальный сервер у провайдера, который арендуется целиком. Значительно более 
широкие возможности. Используется для: 
• AI-агентов и автоматизаций; 
• сервисов с хранением и обработкой данных; 
• ботов и постоянных бэкендов с Docker; 
• всего, что должно работать непрерывно или постоянно ожидать входящих данных. 
Для VPS можно рассмотреть, например, Selectel, Timeweb Cloud, REG.RU или другие аналогичные 
сервисы.

опубликовать, и выбери подходящий тип платформы.

Термины: SSH и SSL/TLS ↘️ 
SSH (Secure Shell) — защищённый способ войти на удалённый сервер и управлять им через 
терминал. Используется при работе с VPS. На обычном хостинге SSH, как правило, не нужен — там 

===== PAGE 42 =====

взаимодействие происходит через панель управления. На managed-платформах SSH тоже не 
нужен: всё загружается через GitHub. 
SSL/TLS-сертификат — когда говорят «поставить SSL», имеют в виду именно TLS-сертификат, 
который обеспечивает работу сайта по протоколу HTTPS. 
HTTPS — защищённый протокол связи между браузером пользователя и сайтом. Без HTTPS 
браузеры показывают предупреждение «Не защищено». Поэтому перед тем, как отправлять сайт 
пользователям, необходимо настроить TLS-сертификат. 
2. Загружаем статический сайт на хостинг Beget 
Шаги деплоя: загружаем статический сайт на хостинг Beget ↘️ 
Мы идём по пути обычного хостинга. В качестве провайдера используем Beget. 
Базовые принципы, которые мы разберём в этом уроке, пригодятся и при работе с VPS: вы 
поймёте, как устроена загрузка проекта, где должны лежать файлы сайта и почему важно не 
публиковать лишние данные. 
Шаг 1. Зарегистрируйтесь в Beget 
Для регистрации понадобятся: 
• мобильный номер; 
• email. 
После регистрации логин и пароль придут на email. Не теряйте это письмо: данные понадобятся 
для входа в панель управления. 
Шаг 2. Создайте сайт в панели управления 
1. Перейдите в раздел «Сайты». 
 
2. Создайте новый сайт с нужным именем. 
3. После создания привяжите к нему технический домен. 


===== PAGE 43 =====

 
Технический домен нужен, чтобы сразу проверить работу сайта без покупки собственного домена. 
Шаг 3. Подготовьте файлы проекта 
Откройте папку с проектом и проверьте, что в ней нет ничего лишнего. 
В папке не должно быть: 
• README-файлов; 
• внутренней документации; 
• .env файлов с ключами; 
• тестовых или служебных файлов, которые не нужны для работы сайта. 
Оставьте только файлы, необходимые для запуска сайта. После этого архивируйте содержимое 
папки в ZIP. 
     Всё, что попадёт в папку public_html, сразу станет доступно всему интернету. Не кладите туда 
конфиденциальные файлы, секретные ключи и служебную документацию. 
Шаг 4. Загрузите проект в файловый менеджер 
1. Перейдите по пути: «Управление» → «Перейти к файлам сайта». 
2. Откройте папку public_html. 
3. Нажмите «Загрузить файлы». 
4. Выберите подготовленный ZIP-архив. 
5. После загрузки нажмите «Распаковать архив». 
6. Путь распаковки оставьте предложенный системой. 


===== PAGE 44 =====

7. После распаковки удалите ZIP-архив. 
       Файловый менеджер также можно открыть напрямую через панель управления: «Хостинг» → 
«Файловый менеджер». 
Шаг 5. Проверьте сайт ↘️ 
Откройте технический домен в браузере. 
Проверьте, что: 
• сайт открывается; 
• картинки отображаются; 
• анимации работают; 
• кнопки и ссылки кликаются; 
• структура страницы не сломалась после загрузки. 
     Ключевое правило: файл index.html должен лежать в корне папки public_html. Если он 
окажется во вложенной папке, сайт не откроется корректно. 
Типичные ошибки при деплое ↘️ 
• Неправильное расположение index.html — файл должен быть в корне `public_html`, а не 
во вложенной папке внутри неё. 
• Некорректные пути к файлам — при локальной разработке пути могут отличаться от тех, 
что нужны на сервере. Пути должны быть написаны относительно корневой директории. 
Если возникла такая ошибка — попроси Claude исправить пути в коде. 
• Конфликт с существующим index.html — если в папке уже лежит старый `index.html`, он 
может конфликтовать с новым. Убедись, что файл один. 
• Ошибки привязки домена или DNS — рассмотрим в следующих видео. 
Обновление сайта в будущем: если нужно внести изменения, самый простой способ — снова 
заархивировать все файлы, удалить старое содержимое из `public_html`, загрузить новый архив и 
распаковать его. Если менялся только один файл — можно перенести только его. 
3. Покупка домена на стороннем регистраторе 
Покупка домена ↘️ 
Домен можно купить как внутри Beget, так и у стороннего регистратора — например, на reg.ru или 
другом аналогичном сервисе. Это полезно знать, если домен уже куплен в другом месте или если 
хочется сравнить цены. 
Процесс покупки: 
1. Вводим желаемое имя домена и нажимаем «Подобрать». 

===== PAGE 45 =====

 
2. Выбираем подходящий вариант из предложенных. 
 
3. Убираем лишние дополнительные услуги, которые предлагает регистратор при 
оформлении заказа, чтобы избежать неожиданных списаний в будущем. 
4. Регистрируемся или входим в аккаунт. 
5. Заполняем контактную информацию — при покупке домена важно указывать реальные 
данные, так как домен оформляется на владельца. 
6. Оплачиваем удобным способом. 
       
Если домен уже куплен на другом сервисе — его можно перенести в Beget через раздел 
«Домены» → «Перенос домена». Либо оставить у текущего регистратора и просто настроить DNS-
записи, указывающие на Beget. 
4. Настройка DNS: связываем домен с хостингом 
Добавление домена в Beget и настройка DNS ↘️ 
После покупки домена нам нужно связать его с хостингом. Если домен куплен у стороннего 
регистратора, это делается через DNS-записи. 
Шаг 1. Добавляем домен в Beget 


===== PAGE 46 =====

В разделе «Домены и поддомены» вводим имя домена и добавляем его в список. После этого 
переходим в «Редактировать DNS» — здесь Beget покажет IP-адрес, который нужно прописать у 
регистратора. 
 
Шаг 2. Добавляем DNS-записи у регистратора 
Переходим в настройки домена на сайте регистратора (например, reg.ru): раздел «DNS-серверы и 
управление зоной» → «Изменить». 
Добавляем две A-записи: 
• Тип: `A`, значение: IP-адрес из Beget, хост: `@` 
• Тип: `A`, значение: тот же IP-адрес, хост: `www` 
Старые записи, которые там были, удаляем — они больше не нужны. 
Шаг 3. Привязываем домен к сайту в Beget 
В разделе «Сайты» находим нужный сайт, нажимаем «Прикрепить домен» и выбираем только что 
добавленный домен. Изменения вступают в силу в течение 10 минут — иногда нужно просто 
подождать. 
     
DNS-записи распространяются не мгновенно. Если сайт не открывается сразу — подожди, почисти 
кэш браузера или проверь в другом браузере. Также убедись, что email при регистрации у 
регистратора подтверждён — без этого домен может не активироваться. 
При помощи DNS мы связали провайдера домена с хостингом. 
5. Привязка домена, проверка сайта и выпуск SSL-сертификата 
Финальная проверка привязки домена ↘️ 
После того как DNS-записи распространились, проверяем, что домен привязан к нужному сайту. 
Если что-то пошло не так (например, создался лишний сайт), домен всегда можно открепить и 
прикрепить заново к нужной папке — домен никуда не пропадёт, он зафиксирован в Beget. 
Когда сайт успешно открывается по домену — переходим к следующему шагу. 
Выпуск SSL/TLS-сертификата ↘️ 
Сайт открывается, но соединение пока работает по HTTP — браузер показывает неприятный 
восклицательный знак «Не защищено». Нам нужно перевести его на HTTPS. 
Как выпустить сертификат в Beget: 
Переходим: «Домены» → «Управление SSL-сертификатами» → выбираем бесплатный Let's 
Encrypt → нажимаем «Установить». 
Заявка отправляется автоматически. Выпуск сертификата для нового домена может занять от 
нескольких часов до двух дней. Информация о статусе придёт на email. 


===== PAGE 47 =====

       
Let's Encrypt — бесплатный TLS-сертификат, которого полностью достаточно для обычного сайта. 
Он доступен на любом хостинге. 
     
Не отправляй сайт пользователям до тех пор, пока SSL-сертификат не выпущен и сайт не 
открывается по HTTPS. 
Итог: что мы сделали ↘️ 
Подведём итог всего процесса деплоя: 
1. Создали сайт в Beget — папка с именем сайта, внутри которой находится `public_html` со 
всеми файлами проекта. 
2. Загрузили файлы через файловый менеджер: архивировали, загрузили, распаковали. 
3. Купили домен — в Beget или у стороннего регистратора. 
4. Настроили DNS — добавили A-записи с IP-адресом Beget на стороне регистратора домена. 
5. Привязали домен к сайту в Beget. 
6. Выпустили SSL-сертификат через Let's Encrypt — ждём активации. 
После активации сертификата восклицательный знак исчезнет и сайт будет работать по 
защищённому протоколу HTTPS. 
6. Базовые принципы безопасности при деплое 
Что не должно попасть в public_html ↘️ 
В папку `public_html` кладём только то, что должен увидеть браузер пользователя: 
•     README и документация 
•     Служебные и системные файлы 
•     `.env` и `.env.local` — файлы с секретными ключами и токенами 
•     Системные промпты агентов 
Серверная логика (например, PHP-бэкенд) может находиться в других папках за пределами 
`public_html`. В публичной папке — исключительно фронтенд. 
     
Всё, что лежит в `public_html`, немедленно становится доступным всему интернету. 
Защита аккаунтов и конфиденциальных данных ↘️ 
• Защищаем паролем доступ к панели Beget и к аккаунту регистратора домена. Если 
доступна двухфакторная аутентификация — включаем её. 
• Не отправляем в чат с ИИ пароли, секретные ключи, платёжные данные и любую 
приватную информацию. Как только данные отправлены — они считаются 
скомпрометированными. В ИИ отправляем только контекст, документацию и общую 
информацию о проекте.

• Понимаем, куда уходят данные форм. Если на лендинге есть форма заявки, нужно чётко 
понимать, кто и как получает эти данные. Если форма ещё не настроена — лучше 
поставить ссылку на соцсеть или мессенджер, чем оставлять заглушку. 
Хорошие привычки после деплоя ↘️ 
• Делаем бэкапы проекта при каждом обновлении. Базовый вариант — создавать папки с 
версиями. Более профессиональный — использовать GitHub: завести аккаунт, 
установить Git и версионировать проект правильно. 
• Проверяем сайт глазами пользователя после каждой публикации: все страницы, формы, 
ссылки, изображения. Убеждаемся, что не выложили лишние файлы. 
• Обязательно подключаем HTTPS перед тем, как делиться сайтом с пользователями. 
       
Чтобы развивать сайт дальше, можно спросить Claude Code: что можно улучшить во фронтенде, 
нужен ли бэкенд и как его написать, какую дополнительную документацию (например, по Beget) 
стоит добавить в контекст. Главное — не передавать секретные ключи и пароли.

• Railway (railway.app) — managed-платформа для деплоя бэкендов, баз данных и 
фронтендов. 
• Netlify (netlify.com) — managed-платформа для деплоя статических сайтов и фронтенда. 
• GitHub (github.com) — сервис для хранения кода и версионирования проектов. 
Рекомендуется для бэкапов и профессиональной работы с кодом. 
• Let's Encrypt — бесплатный центр выдачи TLS-сертификатов. Доступен через панель 
управления большинства хостингов. 
Словарь занятия ↘️ 
• Деплой — процесс переноса проекта с локального компьютера на удалённый сервер, 
после которого сайт или сервис становится доступен всем пользователям интернета. 
• Сервер — компьютер, постоянно подключённый к интернету и готовый обслуживать 
запросы пользователей. 
• Localhost — локальный адрес, по которому проект открывается только на вашем 
компьютере во время разработки. 
• Хостинг — услуга аренды места на сервере для размещения файлов сайта. Подходит для 
статических сайтов, лендингов, PHP-сайтов и CMS. 

===== PAGE 49 =====

• VPS (Virtual Private Server) — виртуальный выделенный сервер, который арендуется 
целиком. Подходит для сложных сервисов, ботов, AI-агентов и всего, что требует 
постоянной работы. 
• Managed-платформа — облачный сервис (Vercel, Railway, Netlify), который берёт на себя 
инфраструктурные задачи: деплой, масштабирование, HTTPS. Обычно работает в связке с 
GitHub. 
• public_html — корневая публичная папка сайта на хостинге. Всё, что в ней лежит, доступно 
пользователям интернета. 
• SSH (Secure Shell) — защищённый протокол для подключения к удалённому серверу и 
управления им через командную строку. Используется при работе с VPS. 
• SSL/TLS-сертификат — цифровой сертификат, обеспечивающий шифрование соединения 
между браузером и сервером. 
• HTTP / HTTPS — протоколы передачи данных в интернете. HTTPS — защищённая версия, 
работающая с TLS-сертификатом. 
• DNS (Domain Name System) — система, которая связывает доменное имя 
(например, mysite.ru) с IP-адресом сервера, где находится сайт. 
• A-запись — тип DNS-записи, которая указывает, на какой IP-адрес должен резолвиться 
домен. 
• Let's Encrypt — бесплатный центр сертификации, выдающий TLS-сертификаты. Доступен на 
большинстве хостингов. 
• GitHub — сервис для хранения кода с системой версионирования Git. Используется для 
бэкапов и командной разработки. 
• .env — файл с переменными окружения, в котором хранятся секретные ключи и токены. 
Никогда не должен попадать в публичную папку или в чат с ИИ. 
• CMS (Content Management System) — система управления контентом (например, 
WordPress). Позволяет управлять сайтом без написания кода.

небольшая часть людей платит за подписку. И лишь около 0,04% населения работает с так 
называемой агентной архитектурой. 
Вы находитесь в числе тех, кто использует агентные решения. Мы с вами действительно в самом 
начале пути — и очень круто, что вы в это запрыгнули.

Ценность навыка появляется не тогда, когда ты просто говоришь «я умею использовать Claude

Правильные формулировки при позиционировании себя: 
• «Довожу прикладную задачу до рабочего результата»

Это продуктовый подход: находим боль — даём конкретное решение. На рынке много запросов, 
где не нужна большая команда разработки и долгий цикл. Малый бизнес сталкивается с 
понятными рабочими проблемами: нужен простой лендинг, внутренний помощник, 
автоматизация контента. 
Общая формула работы: 
1. Понять запрос 
2. Собрать первый вариант — MVP (Minimal Viable Product) 
3. Проверить и доработать 
4. Запустить и объяснить пользу

Форматы услуг после курса ↘️ 
Мы выделяем шесть направлений, в которых можно предлагать свои услуги: 
1. Сайт — упаковка и привлечение. Внутренние сайты, лендинги, страницы услуг, портфолио. 
Людям нужно присутствие в цифровом пространстве — и вы можете закрыть эту потребность. 
2. Внутренний сервис — панели, базы заявок, чек-листы, внутренние автоматизации для бизнеса, 
которые помогают ускорить рутинные рабочие процессы. 
3. ИИ-агент — от простых чат-ботов и помощников по базе данных до сложных мультиагентных 
систем. Позиционирование: архитектор агентных решений. 
4. Автоматизация — личные и бизнес-автоматизации, делегирование задач искусственному 
интеллекту. 
5. Контент — контент-планы, генерация постов и медиафайлов, автоматические контент-
заводы на заказ. 
6. Собственный продукт — SaaS (Software as a Service). Продукт, который монетизируется через 
привлечение частных клиентов или решает повторяющуюся проблему бизнеса. 
     
Во всех этих направлениях мы не продаём Claude Code, агентов или нейросеть. Мы продаём

Пример: один клиент — эксперт по английскому, автор онлайн-курсов или автосервис — может 
одновременно нуждаться в:

• ассистенте для сбора лидов 
• CRM-автоматизации для работы с клиентами

• исследовательском агенте для анализа конкурентов 
• внутренней автоматизации рутинных процессов 
3. Как говорить о навыке на языке пользы 
Как говорить о навыке на языке пользы ↘️ 
Техническая база — это важно. Но когда показываешь проект заказчику, нужно мыслить иначе. 
Не так: «Я подключил агентов» или «Я сделал сайт» — это просто техническая деталь, 
расплывчатая и непонятная клиенту. 
А вот так: 
• «Отчёт собирается быстрее, меньше ручного копипаста, проще проверить 
результат» 
• «У услуги появилась страница, куда можно приводить клиентов и получать заявки» 
Акцент на пользе делает ценность очевидной. 
Структура описания любого проекта: 
• Проблема — что было неудобно, долго, дорого или неструктурно; кто с этим сталкивался и 
как часто; почему это мешало работе 
• Решение — что именно собрано в проекте; какой сценарий проходит пользователь на 
входе и что получает на выходе 
• Результат — где экономия времени, где экономия денег, где снижается ручная рутина, 
повышается качество, скорость, предсказуемость 
Формула кейса: 
«Я сделал проект для [кого], который помогает решить [проблему]. Пользователь делает 
[действие], на выходе получает [результат]. Это полезно потому, что [польза]. Дальше проект 
можно развивать через [шаги].» 
       
Используй эту формулу, чтобы оформить свои кейсы и уверенно идти к новым заказчикам. 
Критерии сильного финального проекта ↘️ 
Проект должен отвечать трём критериям: 
1. Решает конкретную задачу. Понятная проблема или потребность — личная, учебная, для 
работы, для фриланса, для потенциального клиента. Формулировка должна быть простой: кому 
помогает и зачем. 
2. Демонстрирует сценарий. Не просто «проект существует», а показано, что делает человек или 
система на входе и какой результат получается на выходе. 

===== PAGE 58 =====

3. Польза объясняется простыми словами. Любой человек должен понять, почему это лучше, чем 
то, что было раньше. 
     
Если к этому моменту у тебя что-то получилось не до конца или кажется неполноценным — это 
нормально. Мы осваивали базу. Попробуй пересмотреть нужные уроки и довести проект до 
рабочего состояния. Даже простая автоматизация или несколько субагентов — уже настоящий 
кейс.

Примеры реальных проектов ↘️ 
Проект 1: Лендинг. Минималистичный лендинг с информацией об услугах, встроенным видео с

Словарь занятия ↘️ 
• Монетизация навыка — превращение умения работать с инструментом в услугу или 
продукт, за который платят деньги 
• MVP (Minimal Viable Product) — минимально жизнеспособный продукт; первая рабочая 
версия, собранная быстро для проверки гипотезы 
• SaaS (Software as a Service) — программное обеспечение как услуга; продукт, за 
использование которого клиент платит регулярно 
• Агентная архитектура — система, в которой ИИ-агент самостоятельно выполняет задачи, 
вызывает инструменты и делегирует подзадачи субагентам 
• Кодинг-скаффолд (coding scaffold) — архитектура и инструменты вокруг ИИ, с помощью 
которых ведётся разработка 
• Продуктовый подход — методология, при которой сначала находится боль (проблема 
аудитории), затем создаётся конкретное решение под неё 
• Итерационный подход — цикл разработки: собрал MVP → получил обратную связь → 
доработал → показал снова 
• B2B (Business to Business) — направление работы, при котором вы продаёте услуги или 
продукты бизнесу, а не частным лицам 
• Кейс — оформленный пример выполненного проекта: проблема, решение, результат 
• Язык пользы — способ описания своей работы через конкретный эффект для клиента, а не 
через технические детали
